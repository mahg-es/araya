# req-001 — ARAYA command discovery, manual, specialist delegation and mandatory command expertise

Status: new.
Created: 2026-07-21.

## Request

ARAYA dispone de comandos, funciones, skills y agentes especializados que no se utilizan de forma consistente porque los agentes pueden desconocer su existencia, no saber cómo invocarlos o ejecutar directamente tareas que deberían delegar.

Se solicita incorporar una capacidad transversal para que todos los agentes ARAYA, sin excepción:

1. Conozcan qué comandos, funciones, skills y agentes especialistas existen.
2. Sepan cuándo deben utilizarlos.
3. Consulten instrucciones precisas antes de ejecutar una función desconocida.
4. Comprueben si ARAYA ya dispone de una capacidad adecuada antes de implementar lógica nueva.
5. Deleguen el trabajo en el agente especialista competente en lugar de ejecutarlo directamente.
6. Respeten los límites de autoridad, responsabilidad, herramientas y permisos de cada agente.

Debe existir un manual operativo accesible mediante comandos equivalentes a:

```text
/araya:man
/araya:man <comando-o-función>
/araya:man <agente>
/araya:man <skill>
/araya <comando> --help
/araya help
```

El sistema debe permitir:

* listar comandos, funciones, skills y agentes;
* buscar capacidades por nombre, dominio, propósito o tarea;
* mostrar sintaxis, argumentos, opciones y ejemplos;
* explicar cuándo utilizar y cuándo no utilizar cada capacidad;
* indicar qué agente es responsable de cada tipo de trabajo;
* mostrar precondiciones, permisos, riesgos y efectos secundarios;
* identificar capacidades relacionadas o alternativas;
* informar si una función está disponible, deshabilitada, obsoleta o no instalada;
* enlazar cada elemento con su fuente de verdad dentro del repositorio.

Debe crearse una skill transversal obligatoria, con un nombre equivalente a:

```text
araya-command-and-delegation-expert
```

Esta skill debe estar disponible para todos los agentes ARAYA, sin excepción.

La skill debe enseñar a cada agente a:

1. Consultar el catálogo de ARAYA antes de iniciar una tarea.
2. Identificar comandos, funciones, skills y especialistas relevantes.
3. Consultar `--help` o `/araya:man` antes de usar una función desconocida.
4. Preferir capacidades nativas de ARAYA frente a procedimientos manuales duplicados.
5. No inventar comandos, argumentos, flags, agentes o comportamientos.
6. Verificar disponibilidad, compatibilidad y permisos antes de ejecutar.
7. Delegar en el agente especializado cuando exista uno competente.
8. No asumir tareas fuera de su propia autoridad o responsabilidad.
9. Registrar cuando una capacidad o especialista necesario no existe.
10. Proponer nuevas capacidades únicamente después de confirmar que no existe una equivalente.

## Specialist Delegation Contract

La delegación en agentes especialistas es obligatoria cuando ARAYA disponga de un agente competente para la tarea.

Sonia debe:

* analizar el trabajo;
* dividirlo en Workstreams y Atomic Work Units;
* identificar dependencias;
* solicitar a Aurora la elegibilidad de agentes y modelos;
* asignar cada unidad al agente especialista correspondiente;
* coordinar secuencia, paralelismo, validación y cierre;
* verificar que cada especialista devuelve evidencia suficiente.

Sonia no debe ejecutar directamente:

* arquitectura;
* implementación backend o frontend;
* pruebas;
* seguridad;
* infraestructura;
* documentación técnica;
* análisis de datos;
* diseño educativo;
* tareas pertenecientes a cualquier otro especialista disponible.

Sonia puede ejecutar únicamente actividades propias de su autoridad de entrega, como planificación, descomposición, coordinación, seguimiento, resolución de dependencias y consolidación de resultados.

Un agente solamente podrá ejecutar trabajo fuera de su especialidad cuando:

1. No exista un especialista disponible.
2. Aurora confirme la ausencia de una capacidad adecuada.
3. La excepción quede registrada.
4. Los riesgos y límites sean explícitos.
5. La gobernanza vigente permita esa sustitución.

La presión de tiempo, simplicidad aparente o conveniencia no son motivos suficientes para evitar la delegación.

## Context

ARAYA pierde valor cuando sus agentes desconocen sus funciones, no consultan la ayuda disponible o ejecutan directamente tareas asignadas a otros roles.

El caso que origina este requerimiento es que Sonia ejecutó directamente una incorporación que debía haber coordinado y delegado entre agentes especialistas.

La existencia de documentación aislada no garantiza el descubrimiento ni la delegación correcta. La ayuda y la selección de especialistas deben formar parte del flujo normal de cada agente.

El manual no debe ser una copia mantenida manualmente. Debe generarse o validarse contra las fuentes reales del repositorio para evitar divergencias.

Antes de implementar, se debe auditar:

* el registro actual de comandos;
* los mecanismos de ayuda existentes;
* los comandos slash y CLI;
* las funciones del runtime;
* las skills instaladas;
* los agentes, capacidades y límites de autoridad;
* las reglas actuales de delegación;
* la asignación de skills a agentes;
* la documentación existente;
* los casos donde un orquestador ejecutó trabajo de especialistas.

## Acceptance

El requerimiento se considera satisfecho cuando:

1. Existe un catálogo canónico de comandos, funciones, skills y agentes.
2. `/araya:man` lista las capacidades disponibles.
3. `/araya:man <función>` muestra propósito, sintaxis, parámetros, ejemplos, restricciones y especialista responsable.
4. `/araya:man <agente>` muestra responsabilidad, capacidades, skills, permisos y tareas que no debe realizar.
5. Cada comando soportado dispone de `--help` o de una justificación documentada.
6. La ayuda se obtiene desde el registro real o se valida automáticamente contra él.
7. Una función inexistente devuelve un error claro y únicamente sugiere funciones reales.
8. Existe búsqueda por palabras clave, dominio, agente, skill y propósito.
9. Se crea la skill transversal obligatoria.
10. Todos los agentes heredan o tienen asignada explícitamente dicha skill.
11. La validación falla cuando un agente nuevo no recibe la skill.
12. Todo agente consulta capacidades existentes antes de improvisar una solución.
13. Sonia delega arquitectura, desarrollo, pruebas, seguridad, infraestructura y documentación en especialistas.
14. Una prueba falla cuando Sonia u otro orquestador ejecuta directamente trabajo que corresponde a un especialista disponible.
15. Las excepciones de delegación requieren evidencia de que no existe un especialista elegible.
16. Aurora participa en la determinación de elegibilidad cuando la asignación no sea evidente.
17. Los agentes demuestran que pueden descubrir una capacidad no incluida directamente en su prompt.
18. Ningún agente inventa comandos, argumentos, skills o especialistas durante las pruebas.
19. Existe una prueba que confirma que el agente busca una función existente antes de proponer una implementación duplicada.
20. La documentación cubre pi.dev, Codex, Claude CLI, AGY y los adaptadores oficialmente soportados.
21. La solución respeta la frontera entre ARAYA Framework público y ARAYA Core privado.
22. Los cambios incluyen pruebas unitarias, integración, regresión y validación documental.
23. Daneel verifica que el catálogo, las asignaciones y la delegación coinciden con repository truth.
24. La entrega identifica el estado real en workspace, feature branch, dev-mahg, main, release y runtime.
25. La capacidad no se considera entregada hasta demostrar que todos los agentes descubren funciones y delegan correctamente.
